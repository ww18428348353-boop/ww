### 2. 真实 `.pp` 样本：用项目内真实数据反推规则

工具不是凭空设计一套调色格式，而是基于真实项目 `.pp` 文件做逆向和验证。

真实 `.pp` 样本的作用：

- 提供官方工具实际写出的参数结构；
- 确认 `.pp` 文件头、区域数量、参数排列方式；
- 反推出 14 字段参数含义；
- 验证字段默认值和取值范围；
- 保证新工具生成的 `.pp` 能被原有项目接受。

`.pp` 文件结构大致为：

```text
1001 0 255
7
256 0 0
0 256 0
0 0 256
0 500 500 100 128
...
```

含义：

```text
1001：magic 文件标识
0：默认 UI 区域索引
255：全局透明度
7：region 数量
后续每组 14 字段：一个 Region / Skin 调色方案
```

通过真实样本，可以把旧工具里看不见的调色逻辑转化为可读、可测、可扩展的 Python 数据结构。

### 4. `roundtrip_check`：回写严格模式 + 回读验证

截图中 `roundtrip_check` 是整个逆向链路的关键质量保障。

它的含义是：

```text
读取原始 .pp
  ↓
解析成内部数据结构
  ↓
再写回 .pp
  ↓
重新读取写出的 .pp
  ↓
对比前后数据是否一致
```

也就是“读 → 写 → 再读”的闭环校验。

目标：

```text
确保工具不是只能读，而是真的能稳定写回官方兼容格式。
```

严格模式要求：

- 数字顺序一致；
- 字段数量一致；
- region 数量一致；
- 默认值不丢失；
- 写出后可再次解析；
- 回读结果与写出前一致。

这一步解决的是生产风险问题。  
没有回读验证，工具可能看起来能保存，但项目或旧工具打开时异常。  
加入 `roundtrip_check` 后，`.pp` 读写可靠性更高。

### 6. 3×3 通道混合矩阵：随机换色的底层基础

截图中提到的 `3×3通道混合矩阵` 是 PP 调色的第一层核心。

矩阵结构：

```text
rr rg rb
gr gg gb
br bg bb
```

计算方式：

```text
新 R = (rr × R + rg × G + rb × B) / 256
新 G = (gr × R + gg × G + gb × B) / 256
新 B = (br × R + bg × G + bb × B) / 256
```

默认不变矩阵：

```text
256   0   0
  0 256   0
  0   0 256
```

随机换色时，工具不会完全无约束随机，而是控制范围：

```text
主对角线 rr / gg / bb：180～330
副对角线 rg / rb / gr / gb / br / bg：0～80
```

这样做的原因：

- 主对角线保留原始角色识别度；
- 副对角线提供轻量通道串扰；
- 避免颜色完全崩坏；
- 避免生成乱码式配色；
- 更容易得到可用方案。

---

### 7. RGB → HSL → RGB 四步变换

截图中的 `RGB→HSL→RGB 4步变换` 指 PP 调色的第二层处理。

流程为：

```text
输入 RGB
  ↓
RGB 转 HSL
  ↓
调整 H / S / L
  ↓
HSL 转 RGB
  ↓
输出 RGB
```

其中：

```text
H：色相偏移
S：饱和度缩放
L：亮度缩放
```

随机换色中，这一步主要负责整体颜色风格变化。

例如：

```text
红色系 → 蓝色系
暖色系 → 冷色系
低饱和 → 高饱和
暗色 → 明亮
```

比单纯矩阵混合更符合美术对“换色”的直觉。

### 8. HSL Q500 定点后处理与 Q 标定体系

截图中提到：

```text
HSL Q500 定点后处理
Q 标定体系
```

这是官方 PP 参数的定点数值体系。

HSL 参数范围：

```text
h：0～500
s：0～1000
l：0～1000
```

其中：

```text
h = 500 等价于 360° 色相旋转
s = 500 等价于 100% 饱和度
l = 500 等价于 100% 亮度
```

换算方式：

```text
H_new = H + h / 500 × 360°
S_new = S × s / 500
L_new = L × l / 500
```

线亮度和对比度也有自己的 Q 标定：

```text
lb：Q100，100 = 100% 线亮度
c：Q128，128 = 100% 对比度
```

对应公式：

```text
RGB = RGB × lb / 100
RGB = (RGB - 128) × c / 128 + 128
```

这套 Q 标定体系的意义是：

- 和官方工具数值体系保持一致；
- 保存到 `.pp` 后不需要额外转换；
- 美术看到的参数和实际文件参数一致；
- 随机生成时可控制在合理范围内。

### 9. 笔线：保护角色边缘和像素风格识别

截图中提到“笔线”，可以理解为 2D 造型资源中的轮廓线、描边线、暗部线条和关键结构线。

PP 随机换色不能只追求颜色变化，还要保证：

```text
轮廓线不能丢
暗部结构不能糊
角色识别度不能消失
像素边缘不能过曝
```

因此随机参数设计会避免：

- 亮度过高导致笔线发白；
- 对比度过低导致轮廓糊掉；
- 饱和度过低导致整体发灰；
- 通道矩阵过激导致暗线变脏；
- 颜色反差过大导致角色材质失真。

这也是随机范围不做全域随机的原因。  
工具追求的是“快速得到可展示方案”，不是生成不可控噪声。

### 10. 随机变色：从手动试色升级为自动探索

截图核心功能是“随机变色”。

随机变色的设计目标：

```text
随机生成，但尽量可用
```

不是完全随机 14 个字段，而是分组控制：

| 参数类型 | 随机范围 | 目的 |
|---|---:|---|
| 主对角线 `rr / gg / bb` | 180～330 | 保留原角色结构 |
| 副对角线 | 0～80 | 轻量混色，制造变化 |
| 色相 `h` | 0～500 | 覆盖完整色相环 |
| 饱和度 `s` | 350～700 | 避免过灰或过艳 |
| 亮度 `l` | 400～600 | 避免全黑或过曝 |
| 线亮度 `lb` | 80～140 | 保留画面层次 |
| 对比度 `c` | 100～180 | 保留笔线和细节 |

使用方式：

```text
点击随机换色
  ↓
生成一组 PP 参数
  ↓
实时预览 PNG / TGA 或 TCP 解码图
  ↓
满意后复制为新方案
  ↓
继续微调
  ↓
保存 .pp 或导出染色图
```

它解决的是美术探索效率问题：

```text
过去：从 0 拖 14 个参数，慢慢找方向
现在：随机生成多个候选，再挑好的微调
```

## 背景

### 1. 传统 PP 变色流程成本高，效率低

当前 2D 角色、NPC、造型、服饰等资源在做变色时，传统流程主要依赖官方旧工具或人工调参。

这类资源经常需要做多套配色，例如：

- 普通款；
- 稀有款；
- 节日款；
- 活动款；
- 染色款；
- 玩家 DIY 款；
- 同造型多皮肤版本。

但旧流程整体比较割裂，调色、预览、保存、复用都不够顺畅。

### 2. 官方旧工具封闭，难以支撑批量化生产

官方旧工具虽然能做区域变色，但使用体验偏传统，主要问题包括：

- 工具是老式 GUI，不能很好接入当前批量流程；
- 没有实时高效的多方案管理；
- 一个 `.pp` 文件里的多个区域 / 多套方案不方便统一维护；
- 无法方便做随机换色、快速试色；
- 自动播放、动作、方向、帧率、背景预览能力不足；
- 不方便和当前项目内的 TGA 序列、TCP 资源流程联动；
- 算法不可控，无法直接扩展新功能。

### 3. 多方向、多帧资源调色更容易返工

在实际生产中，美术经常需要基于同一套造型快速调整出多个颜色方向。

如果每一套都手动拖滑块、保存、对比、再回头微调，效率很低。尤其是角色或服饰有多个方向、多帧动画时，只看静态单帧很容易漏掉动态表现问题。

可能出现的问题包括：

- 某一帧颜色突兀；
- 某个方向颜色不协调；
- 轮廓边缘脏色明显；
- 透明区域异常；
- 暗部细节丢失；
- 高光过曝；
- 动态播放时整体观感不稳定。

### 4. 需要一套现代化 PP 变色管线

所以该模块的核心价值，是把“官方 PP 变色流程”重新工具化、可视化、批量化。

它不是简单调用老工具，而是把 `.pp` 调色算法、`.pp` 文件读写、校验逻辑、多方案管理、实时预览都重新实现，形成一套可控、可扩展的新管线。

## 目标

核心目标是：**在保持官方 PP 变色结果兼容性的前提下，把旧的单点调色流程升级为实时预览、多方案管理、随机换色、可复用的现代化调色工具。**

### 1. 复刻官方 PP 变色逻辑

工具重新实现官方区域变色的 14 字段调色算法，确保输出 `.pp` 文件能被原有项目和官方流程识别。

### 2. 脱离旧工具限制

不再依赖老 GUI 工具手动操作，调色核心由 Python + NumPy 实现，方便调试、扩展、批量处理和集成。

### 3. 提升调色效率

支持拖动滑块实时刷新预览，减少“调一次、保存一次、打开看一次”的反复操作。

### 4. 支持多方案管理

一个 `.pp` 工程可以维护多套变色方案，每套方案对应一个 Region / Skin，方便复制、重命名、对比和保存。

### 5. 支持随机换色

工具内置合理区间随机参数，不是完全乱随机，而是生成“像换肤而不是乱码”的可用配色，帮助美术快速找到方向。

### 6. 支持 TCP / TGA 资源预览

兼容 `.tcp` 文件和 TGA 序列。TCP 通过官方解码器转成 TGA 后预览；TGA 序列则可以直接加载并实时变色。

### 7. 支持动态动画预览

提供方向、帧率、播放 / 暂停、背景色、背景图等控制，让美术能直接看变色后动态效果。

### 8. 沉淀标准化 PP 变色管线

后续可扩展到批量导出、自动生成多套染色方案、造型资源批量适配等生产环节。

## 解决方案

### 1. 重构 PP 官方变色管线，摆脱旧工具限制

该工具对标的是“大话造型区域变色配色工具”，但不是简单调起旧工具，而是重新实现核心算法。

整体流程可以理解为：

```text
TCP / TGA 资源
  ↓
加载素材
  ↓
读取或新建 .pp 调色方案
  ↓
14 字段 PP 参数调色
  ↓
实时预览角色 / 造型 / 服饰效果
  ↓
多方案管理 / 随机换色 / 手动微调
  ↓
保存为项目可用 .pp 文件
```

其中：

- TCP 解码仍使用稳定的官方 `tcp_2_tga.exe`；
- 颜色变换算法完全由工具自身实现；
- `.pp` 文件读写完全由工具自身实现；
- 多方案管理、随机换色、动态预览是新增能力。

### 2. PP 14 字段调色面板

工具完全复刻官方 PP 的 14 字段调色结构。

一个 PP 调色方案由 14 个整数参数组成：

```text
rr rg rb
gr gg gb
br bg bb
h s l
lb
c
```

可以分为三大类：

---

## 一、通道混合矩阵

通道混合矩阵共 9 个参数：

| 参数 | 含义 | 默认值 |
|---|---|---:|
| `rr` | 红通道中的红色比例 | 256 |
| `rg` | 红通道中的绿色混合 | 0 |
| `rb` | 红通道中的蓝色混合 | 0 |
| `gr` | 绿通道中的红色混合 | 0 |
| `gg` | 绿通道中的绿色比例 | 256 |
| `gb` | 绿通道中的蓝色混合 | 0 |
| `br` | 蓝通道中的红色混合 | 0 |
| `bg` | 蓝通道中的绿色混合 | 0 |
| `bb` | 蓝通道中的蓝色比例 | 256 |

默认矩阵：

```text
256   0   0
  0 256   0
  0   0 256
```

表示颜色不变。

实际计算逻辑：

```text
新 R = (rr × R + rg × G + rb × B) / 256
新 G = (gr × R + gg × G + gb × B) / 256
新 B = (br × R + bg × G + bb × B) / 256
```

这个矩阵可以实现比较底层的颜色重组。例如：

```text
增加 rg：把绿色信息混入红色通道
增加 gb：把蓝色信息混入绿色通道
降低 rr：减少原始红色强度
```

它比普通 HSL 更底层，可以做出更明显的换色效果。

---

## 二、HSL 后处理

HSL 部分有 3 个参数：

| 参数 | 含义 | 范围 | 默认值 |
|---|---|---:|---:|
| `h` | 色相偏移 | 0～500 | 0 |
| `s` | 饱和度 | 0～1000 | 500 |
| `l` | 亮度 | 0～1000 | 500 |

其中：

```text
h = 500 相当于 360° 色相循环
s = 500 相当于 100% 饱和度
l = 500 相当于 100% 亮度
```

换算逻辑：

```text
H = H + h / 500 × 360°
S = S × s / 500
L = L × l / 500
```

也就是说：

```text
s < 500：降低饱和度
s > 500：提高饱和度

l < 500：压暗
l > 500：提亮
```

---

## 三、线亮度与对比度

最后两个参数：

| 参数 | 含义 | 范围 | 默认值 |
|---|---|---:|---:|
| `lb` | 线亮度 | 0～200 | 100 |
| `c` | 对比度 | 0～256 | 128 |

线亮度公式：

```text
RGB = RGB × lb / 100
```

对比度公式：

```text
RGB = (RGB - 128) × c / 128 + 128
```

效果：

```text
lb > 100：整体更亮
lb < 100：整体更暗

c > 128：对比更强
c < 128：对比更弱
```

## 3. `.pp` 文件读写能力

`.pp` 文件是官方变色配置文件。工具已经逆向并重新实现读写。

`.pp` 文件本质是文本格式，结构大致如下：

```text
1001 0 255
```


每个 Region 对应一组 14 字段参数。

- 修改后写回 `.pp`；
- 保持官方格式兼容；
- 输出能被旧工具和项目流程识别的文件。

## 5. 随机换色能力

该工具新增了“随机换色”功能。

它不是完全随机，因为完全随机很容易生成不可用颜色，比如：

```text
过曝
全黑
严重偏灰
颜色脏
通道混乱
角色识别度消失
```

所以工具采用“合理区间随机”。

随机规则大致如下：

| 参数类型 | 随机范围 | 目的 |
|---|---:|---|
| 主对角线 `rr / gg / bb` | 180～330 | 保留原始造型识别度 |
| 副对角线 | 0～80 | 轻量通道混合，避免颜色崩坏 |
| 色相 `h` | 0～500 | 允许全色相变化 |
| 饱和度 `s` | 350～700 | 避免过灰或过艳 |
| 亮度 `l` | 400～600 | 避免全黑或过曝 |
| 线亮度 `lb` | 80～140 | 保留整体层次 |
| 对比度 `c` | 100～180 | 保留细节和材质感 |

使用方式：

```text
点击随机换色
  ↓
工具生成一组合理参数
  ↓
实时刷新预览
  ↓
满意后复制为新方案
  ↓
再手动微调
  ↓
保存 .pp
```

这个功能适合美术快速找方向。不需要从 0 开始拖 14 个滑块，可以先随机出 10～20 个候选，再挑一个继续优化。

## 8. TCP 解码缓存

TCP 解码依赖官方 `tcp_2_tga.exe`，但解码不是热路径。

工具会使用缓存机制：

```text
_tcp_cache/<sha1>/
```

同一个 TCP 文件解码一次后，下次直接走缓存。

好处：

```text
第一次加载：0.2～2 秒
第二次加载：几乎 0 秒
```

并且用户可以一键清空 TGA 缓存，避免缓存占用过多空间。

工具底层使用 NumPy 做向量化运算。调色不是逐像素 Python 循环，而是数组批处理。

| 操作 | 性能 |

| TCP 缓存后加载 | 接近 0 ms |
| 256 色调色板路径 | 小于 0.1 ms |

这意味着拖动滑块时基本可以做到实时反馈。

`PP官方变色` 工具，本质是把旧的官方 PP 变色能力重新实现并升级为现代化生产管线。

它解决的问题不是单纯“换颜色”，而是完整覆盖：

```text
TCP / TGA 素材加载
官方 PP 14 字段调色
.pp 文件读写
多方案 Skin 管理
随机换色
实时预览
方向 / 帧率 / 背景控制
动态效果检查
后续批量导出基础
```

核心价值：

```text
兼容官方格式
算法完全可控
调色实时反馈
多方案统一管理
随机生成候选
降低美术试错成本
为批量变色打基础
```

对于需要大量 2D 造型、服饰、NPC、怪物、活动皮肤、玩家 DIY 染色方案的项目，这个工具能明显提升生产效率和资源复用率。

如果说旧流程是“手动调一套颜色”，那这个工具就是“建立一套可复用、可扩展、可批量化的 PP 变色生产管线”。
