# 角色整体色调变换的技术实现方案研究

## 要点

- **HSL/HSV色相旋转**是最基础的换色方案，通过在色彩空间中偏移H分量实现整体色调变换，但可能改变明暗/冷暖关系
- **GPU Shader方案**（Unity/Cocos/WebGL）在fragment shader中执行RGB↔HSV转换，性能优异且精确可控，是实时渲染的首选
- **通道混合染色系统**是工业级最优方案，使用Mask贴图分区 + RGB矩阵运算，能完美保持原始明暗和冷暖关系不变
- **CSS `hue-rotate()`存在严重局限性**——它实际执行的是RGB矩阵运算而非真正的HSL转换，无法保持饱和度和亮度
- **无彩色系（黑白银灰）需要特殊处理**——饱和度为0的颜色色相旋转完全无效，必须通过去饱和度+明度调整的方式实现
- **色相环角度对应关系明确**：红0°、橙30°、黄60°、绿120°、青180°、蓝240°、紫270°、粉330°~350°，可精确定位目标色系

---

## 概述

在游戏和应用开发中，角色换色（Recolor/Skin Variation）是一种常见需求：给定角色的默认配色，通过点击不同色系按钮（红、橙、黄、绿、青、蓝、紫、粉、黑、白、银、灰），将角色整体色调进行统一变换。这不同于简单的单色替换，而是类似于Photoshop中"色相/饱和度"的整体偏移操作。

本报告系统研究了实现这一功能的多种技术方案，包括HSL/HSV色相旋转、Shader着色器、Canvas像素操作、调色盘替换等，并深入分析了色调映射规则、算法实现细节，以及黑白银灰等无彩色系的特殊处理方式。

## 详细分析

### 一、HSL/HSV色相旋转原理

#### 1.1 色彩空间基础

HSV（Hue, Saturation, Value）是RGB的一种变形，用人类更为熟悉的方式封装了关于颜色的信息 [1]。在HSL和HSV色彩空间中，H（Hue）代表色相，取值范围是0°到360°的圆心角，分布在一个平面的色相环上 [2]。

色相旋转的核心思想是：**将像素从RGB转换为HSV/HSL，修改H分量（加上偏移值），再转换回RGB**。

| 色彩空间 | 分量 | 适用场景 | 特点 |
|---------|------|---------|------|
| RGB | 红、绿、蓝 | 显示设备 | 机器友好，不直观 |
| HSV | 色相、饱和度、明度 | 美术调色 | 直观，方便分别调整 |
| HSL | 色相、饱和度、亮度 | Web开发 | 与HSV类似，L的定义不同 |

#### 1.2 色相偏移的核心操作

在Shader中实现色相偏移只需一行核心代码 [3]：

```glsl
hsv.x = fract(hsv.x + hueShift); // hueShift范围0~1，对应0°~360°
```

HSV染色是目前最为主流的染色方法，美术调整颜色时大部分都是调整的HSV，因为可以方便地分别调整色相、饱和度和明度 [4]。

### 三、Shader着色器方案（推荐）

#### 3.1 RGB与HSV转换的GLSL实现

在GLSL/Shader中，RGB与HSV的转换使用以下标准函数 [3]：

**RGB转HSV：**
```glsl
vec3 rgb2hsv(vec3 c) {
    vec4 K = vec4(0.0, -1.0/3.0, 2.0/3.0, -1.0);
    vec4 p = mix(vec4(c.bg, K.wz), vec4(c.gb, K.xy), step(c.b, c.g));
    vec4 q = mix(vec4(p.xyw, c.r), vec4(c.r, p.yzx), step(p.x, c.r));
    float d = q.x - min(q.w, q.y);
    float e = 1.0e-10;
    return vec3(abs(q.z + (q.w - q.y) / (6.0 * d + e)), d / (q.x + e), q.x);
}
```

**HSV转RGB：**
```glsl
vec3 hsv2rgb(vec3 c) {
    vec4 K = vec4(1.0, 2.0/3.0, 1.0/3.0, 3.0);
    vec3 p = abs(fract(c.xxx + K.xyz) * 6.0 - K.www);
    return c.z * mix(K.xxx, clamp(p - K.xxx, 0.0, 1.0), c.y);
}
```

#### 3.2 Unity中的角色换色方案

Unity中主要有三种换色方法：

**方法一：HSV调色Shader** —— 在片元着色器中将RGB转换为HSV，调整H/S/V各分量后转回RGB [6]。

**方法二：Mask分区染色** —— 使用一张mask贴图的四通道（RGBA）划分染色区域，每个通道标记不同的可染色区域，对每个区域分别应用目标色 [7]。

**方法三：通道混合染色系统（最优）** —— 解决了HSV染色后明暗/冷暖关系改变的问题，基于RGB的颜色叠加方式进行调整，通过原始颜色的通道作为选区和调整基准值 [5]。该方案基于矩阵运算，性能比在Shader中做RGB→HSV→RGB的转换还要好 [5]。

#### 3.3 Cocos Creator色相转换

在Cocos Creator中，色相转换的方法是在片元着色器中采样当前点颜色，将RGB转换为HSV，判定色相区间并加偏移值实现目标色转换 [8]。Cocos Creator提供了企业级Shader开发环境，支持自定义2D精灵着色器 [9]。

### 四、Canvas 2D像素操作方案

#### 4.1 基本实现

Canvas通过ImageData对象操纵像素数据 [13]，基本流程为：

```javascript
const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);
const data = imageData.data;

for (let i = 0; i < data.length; i += 4) {
    let r = data[i], g = data[i+1], b = data[i+2];
    // RGB → HSL 转换
    let [h, s, l] = rgbToHsl(r, g, b);
    // 色相偏移
    h = (h + hueShift) % 360;
    // HSL → RGB 转换
    let [nr, ng, nb] = hslToRgb(h, s, l);
    data[i] = nr; data[i+1] = ng; data[i+2] = nb;
}
ctx.putImageData(imageData, 0, 0);
```

#### 4.2 Konva.js现成方案

Konva.js提供了直接可用的HSL滤镜 [15]：

```javascript
image.filters([Konva.Filters.HSL]);
image.hue(90);         // 色相偏移90°
image.saturation(0.5); // 饱和度调整
image.luminance(0.1);  // 亮度调整
```

### 五、CSS hue-rotate的局限性

CSS `filter: hue-rotate()` 应用色相旋转，通过角度值设定色环旋转幅度 [16]，但存在严重局限性：

> **`hue-rotate()` 被指定为对RGB颜色执行的矩阵运算。它实际上不会将颜色转换为HSL模型，因为HSL是一种非线性操作。因此，它可能无法保持原始颜色的饱和度和亮度** [17]。

| 问题 | 描述 |
|------|------|
| 精度差 | 无法精准指定RGB值，只适合快速改色场景 [18] |
| 全局作用 | 对所有颜色同时旋转，无法针对特定区域 |
| 饱和度/亮度失真 | 不能保持原始颜色的饱和度和亮度不变 |
| 无彩色异常 | 黑白灰等无彩色会被错误地加入颜色 |
| 非真实HSL转换 | 实际是矩阵运算近似，非精确HSL操作 |

张鑫旭指出，虽然`hue-rotate()`可以规避传统按钮赋色中代码量大的缺点，但色值之间饱和度亮度存在差异问题 [19]。

### 六、目标色系的色相角度对应值

根据HSL/HSV色彩空间定义，色环上主要颜色的角度映射如下 [26][27][28]：

| 目标色系 | 色相角度(H) | 饱和度(S) | 明度(V/L) | 备注 |
|---------|------------|-----------|-----------|------|
| 红色 | 0°(360°) | 保持/100% | 保持 | 色环起点 |
| 橙色 | 30° | 保持/100% | 保持 | 红黄之间 |
| 黄色 | 60° | 保持/100% | 保持 | 第一个二次色 |
| 绿色 | 120° | 保持/100% | 保持 | 三等分点 |
| 青色 | 180° | 保持/100% | 保持 | 绿蓝之间 |
| 蓝色 | 240° | 保持/100% | 保持 | 三等分点 |
| 紫色 | 270° | 保持/100% | 保持 | 蓝品红之间 |
| 粉色 | 330°~350° | 降低至50~70% | 提高 | 偏红低饱和 [29] |
| **黑色** | 任意 | → 0% | → 0~10% | 无彩色，特殊处理 |
| **白色** | 任意 | → 0% | → 90~100% | 无彩色，特殊处理 |
| **银色** | 任意 | → 0~5% | → 70~80% | 微暖灰可加微量S |
| **灰色** | 任意 | → 0% | → 40~60% | 无彩色，特殊处理 |

六大主色按照60°圆心角间隔均匀分布：0°红、60°黄、120°绿、180°青、240°蓝、300°洋红 [28]。

### 七、保持明暗关系的换色技术

HSV染色存在一个关键问题：**染色后明暗/冷暖关系可能发生改变** [5]。解决方案包括：

#### 7.1 通道混合算法（推荐）

通道混合算法的核心优势是完美保持明暗关系。当RGB通道内的混合分量百分比之和为100%时，中性灰的输出值等于输入值，从而保持明暗关系不变 [23]。

#### 7.2 亮度保持策略

在Shader中保持明度不变的方法 [24]：

1. **提取亮度**：`L = 0.2126*R + 0.7152*G + 0.0722*B`
2. **仅修改色相/饱和度**：在HSV空间中只改变H和S，保持V不变
3. **分离再合成**：将原图分离为明度图和色相图，变换色相后与原始明度重新合成

---

### 八、无彩色系（黑白银灰）特殊处理

#### 8.1 核心问题

**饱和度为0的颜色为无彩色，就是黑、白、灰** [30]。色相旋转对它们**完全无效**，因为没有色彩成分可以旋转。

#### 8.2 处理方案对比

| 方案 | 实现方式 | 适用场景 | 效果 |
|------|---------|---------|------|
| 去饱和+明度调整 | S→0, L→目标灰度 | 有色→无色 | 自然 |
| 亮度公式去色 | Gray = 0.2126R + 0.7152G + 0.0722B | 保留明暗层次 | 优秀 |
| 饱和度阈值判断 | Shader中检测S值 | 分别处理有色/无色 | 灵活 |
| 着色叠加 | 无色→有色，叠加目标色 | 无色变有色 | 可控 |

#### 8.3 Shader中的条件处理示例

```glsl
vec3 hsv = rgb2hsv(color.rgb);

if (targetIsChromaticColor) {
    // 目标是有彩色：执行色相旋转
    hsv.x = targetHue / 360.0;
    // 可选：调整饱和度
    hsv.y = mix(hsv.y, targetSaturation, strength);
    color.rgb = hsv2rgb(hsv);
} else {
    // 目标是无彩色（黑白银灰）
    float luminance = dot(color.rgb, vec3(0.2126, 0.7152, 0.0722));
    // 保持明暗层次，整体偏移至目标灰度
    color.rgb = vec3(luminance * targetBrightness);
}
```

纯红色饱和度降为0后会变为中等灰色（约50%明度灰），因为HSL中纯红的亮度约为50% [32]。

## 调查笔记

### 调色盘替换（Palette Swap）深度分析

调色盘替换是2D游戏中的经典技术，核心步骤包括 [22]：

1. **颜色划分聚类**：对原始纹理贴图聚类输出K个色块（标签色）
2. **基于聚类结果ReColor**：基于标签色与替换色对进行映射

实现方式包括 [11]：
- 使用索引颜色的精灵图，运行时替换调色盘
- 使用查找纹理（LUT）将源颜色映射到目标颜色
- 在Fragment Shader中根据颜色索引在调色盘纹理中采样

GameMaker的ColorMod方案以每行表面颜色作为查找表绑定到sampler，在着色器中进行快速调色盘交换 [12]。

### 色调映射与线性空间

色调映射（Tonemapping）主要用于HDR→LDR的转换 [20]。在游戏实时渲染中，颜色计算在线性空间进行，输出时需要伽马校正 [21]。这一思想可以借鉴到换色系统中：通过映射函数将一种颜色范围变换到另一种颜色范围。

### 粉色的特殊性

粉色并非色环上的一个纯色点，而是红色区域降低饱和度、提高亮度的结果。标准粉色#FFC0CB的色调角度约为349.5度，饱和度100%，亮度87.6% [29]。实现粉色换色时，不仅需要设置色相到330°~350°范围，还需要降低饱和度并提高亮度。
