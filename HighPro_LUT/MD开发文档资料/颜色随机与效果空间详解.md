# HighPro_LUT — 颜色随机逻辑与效果空间详解

> 版本: v1.0 | 生成日期: 2026-06-02 | 项目根: `x:\Lut变色\HighPro_LUT`

---

## 一、开发环境概要

本项目 HighPro_LUT 是一款高性能序列帧角色换色工具, 以 Windows 桌面 EXE 形式发布. 开发机运行 Windows 11 Pro 24H2 (Build 26100.6899, x64), 配备 Intel Core i9-13900K CPU、64 GB DDR5 内存和 NVIDIA GeForce RTX 4090 独立显卡. 实际最低运行要求为任意支持 D3D 11.0 Feature Level 11_0 的 GPU 加 8 GB 内存.

构建工具链方面, 项目使用 Visual Studio 2026 Community (cl 19.51.36243, x64) 作为 C++ 编译器, 语言标准为 C++17, 编译选项包括 `/utf-8`、`/MP` 多核并行、`/permissive-` 严格模式等. 构建系统为 CMake 3.30.5 配合 Ninja 1.12.1 生成器. UI 框架使用 Qt 6.11.1 msvc2022_64 动态版, 仅依赖 Core、Widgets、Gui 三个模块. Windows SDK 版本为 10.0.26100.0, 提供 D3D11 头文件.

第三方库全部为 header-only, 放在 `extern/` 目录下: stb_image v2.30 负责 TGA/PNG 解码, stb_image_write 负责 PNG 导出, nlohmann/json 3.12.0 负责工程文件的 JSON 序列化. 系统层面链接了 `d3d11`、`dxgi`、`d3dcompiler`、`dxguid` 四个 Windows 自带库, 用于 Direct3D 11 设备管理和运行时 HLSL 着色器编译.

EXE 输出直接落在项目根目录 `HighPro_LUT.exe`, 通过 `build.bat` 一键构建, 内部会自动调用 `vcvars64.bat` 进入 MSVC 环境, 运行 CMake configure + build, 最后用 `windeployqt` 部署 Qt 运行时 DLL (因中文路径兼容性问题, 部署过程借 `%TEMP%` 目录中转).

## 二、效果空间总览

HighPro_LUT 的核心变色能力建立在一套 **7 效果链模型** 之上, 该模型严格对齐 Adobe After Effects 的标准调色效果, 处理顺序固定不可更改. 这 7 个效果按序为:

1. **色相/饱和度** (HSL) — 在 HSV/HSB 色彩空间内旋转色相、调整饱和度和亮度
2. **亮度/对比度** (BrtCtr) — AE 经典传统模式, 公式为 `c = (c - 0.5) * (1 + ctr) + 0.5 + brt`
3. **曲线** (Curves) — 支持 Master 主曲线 + R/G/B 三通道独立曲线, 每条最多 8 个控制点
4. **通道混合器** (ChMix) — 3x4 仿射矩阵, 可做通道交叉映射, 含单色模式
5. **颜色平衡** (ColorBal) — 阴影/中间调/高光三区独立调整, 可选保持亮度
6. **照片滤镜** (PhotoFilter) — 内置 19 种 AE 预设滤镜 (暖色/冷色/纯色等), 可调浓度
7. **自然饱和度** (Vibrance) — 对低饱和区域加强、高饱和区域抑制, 附带均匀饱和度补偿

每个效果都有独立的启用/禁用开关. 7 个开关加参数组成一个 `EffectStack` 结构体, 每个层 (layer) 在每个方案 (scheme) 下各自持有一份独立的 `EffectStack`. 此外还有一个不属于 7 效果开关体系的 **影子保护** 机制: 当源像素亮度低于阈值 (默认 8/255) 时, 输出颜色会线性插值回原色, 防止极暗区域被效果链染上不自然的颜色.

整个效果链在 GPU 端由 `effect_chain.hlsl` 的 Pixel Shader 执行, CPU 端的参数模型定义在 `src/core/ColorEffect.h` 的 `EffectStack` 结构体中, 两者通过 Constant Buffer (cbuffer EffectCB, register b1) 严格对应.

## 三、各效果参数详解

### 3.1 色相/饱和度 (HslParams)

色相参数 `hue` 范围 -180 到 180 度, 传入 shader 时除以 360 得到归一化旋转量. 饱和度 `saturation` 范围 -100 到 100, 其中 -100 表示完全灰阶、+100 表示 2 倍饱和. 亮度 `lightness` 范围 -100 到 100. 该效果使用 HSV (也称 HSB) 色彩模型而非 HSL, 与 AE 行为一致. 在 shader 中, 饱和度的加性逻辑为: 正值时 `S = clamp(S + (1-S)*ds, 0, 1)` (向 1 靠拢), 负值时 `S = clamp(S * (1+ds), 0, 1)` (向 0 靠拢). 亮度 (V) 同理.

### 3.2 亮度/对比度 (BrtCtrParams)

亮度 `brightness` 范围 -150 到 150, 传入 shader 时除以 255. 对比度 `contrast` 范围 -100 到 100, 传入时除以 100. 计算公式为 AE 经典模式: `output = (input - 0.5) * (1.0 + contrast) + 0.5 + brightness`, 结果 saturate 到 [0,1].

### 3.3 曲线 (CurveParams)

包含 4 条曲线: Master 主曲线和 R/G/B 三通道曲线. 每条曲线由一组 (x, y) 控制点定义, x 和 y 均在 0 到 255 范围内, 默认两点 `{(0,0), (255,255)}` 即恒等映射. 控制点通过 `CurveSolver` 使用 Fritsch-Carlson 单调三次 Hermite 插值算法编译为 256 字节的查找表. 4 条曲线打包成一张 256x1 的 R8G8B8A8 纹理, 其中 R 通道存 Master、G 通道存 R 曲线、B 通道存 G 曲线、A 通道存 B 曲线. Shader 采样时先对每个通道查各自的子曲线, 再叠加 Master 曲线.

### 3.4 通道混合器 (ChMixParams)

定义一个 3x4 仿射变换矩阵, 输出 `R' = rr*R + rg*G + rb*B + r_const`, G 和 B 行类似. 参数单位为百分比, 默认单位阵 (rr=gg=bb=100, 其余 0). 支持单色 (monochrome) 模式, 此时用 R 行系数生成灰度值. 参数范围 -200 到 200 (UI 滑块). 在随机逻辑中, 通道混合器被标记为"不开" — 因为容易产生怪异颜色, 不适合写实游戏场景.

### 3.5 颜色平衡 (ColorBalParams)

三个色调区域 (阴影 sR/sG/sB、中间调 mR/mG/mB、高光 hR/hG/hB) 各有 3 个参数, 范围 -100 到 100. Shader 中使用 smoothstep 风格的三区权重: 阴影权重 `wS = max(0, 1 - 2*Y)`, 高光权重 `wH = max(0, 2*Y - 1)`, 中间调权重 `wM = 1 - wS - wH`, 其中 Y 为 Rec.601 亮度 `dot(c, (0.299, 0.587, 0.114))`. 最终加成量 = `(sBal*wS + mBal*wM + hBal*wH) * 0.5`. 可选保持亮度: 变换后按原始亮度与新亮度的比值缩放 RGB.

### 3.6 照片滤镜 (PhotoFilterParams)

内置 19 种 AE 预设, 包括暖色 85/LBA/81、冷色 80/LBB/82、红/橙红/黄/绿/青/蓝/紫/品红/棕褐/深蓝/翡翠绿/深黄/水下. 每个预设定义一个 sRGB 颜色 (filterR/filterG/filterB, 0-255). 浓度 `density` 范围 0-100%, 计算公式为 `output = lerp(input, filterColor, density)`. 可选保持亮度, 逻辑同颜色平衡.

### 3.7 自然饱和度 (VibranceParams)

两个参数: `vibrance` (-100 到 100) 和 `saturation` (-100 到 100). 区别在于 vibrance 对低饱和像素加得多、高饱和像素加得少, 计算公式为 `factor = 1.0 + sat + vib * (1.0 - curSat)`, 其中 `curSat = max(R,G,B) - min(R,G,B)`. 输出 `= Y + (c - Y) * factor`.

### 3.8 影子保护

不属于 7 效果开关, 始终生效. 参数 `shadowProtectThreshold` 默认为 8 (范围 0-255, 0 = 关闭). 当源像素亮度 < 阈值/255 时, 效果链输出与原色做线性插值: `k = lum0 / threshold`, `output = lerp(original, effected, k)`. 确保纯黑及极暗像素不被效果链意外染色.

## 四、颜色随机逻辑 (randomizeStack)

颜色随机的核心函数是 `randomizeStack(EffectStack& s, quint32 seed)`, 定义在 `src/core/ColorEffect.cpp`. 该函数对传入的 `EffectStack` 应用一组随机参数, 设计目标是产出适合写实游戏角色的配色方案 — 大部分情况偏低饱和、接近真实服装质感, 偶发鲜艳创意色.

### 4.1 随机数引擎

使用 `std::mt19937` 梅森旋转引擎. 若 `seed != 0` 则用固定种子 (可用于"同一方案不同层一致性"场景), 否则从 `QRandomGenerator::global()->generate()` 取全局随机种子. 提供两个辅助 lambda: `chance(pct)` 判断百分比概率是否命中, `rint(lo, hi)` 生成闭区间均匀整数.

### 4.2 风格档位系统

随机过程首先掷骰决定一个"风格档位", 概率分布如下:

- **Realistic (写实)** — 70% 概率. 微调色相, 饱和度大幅下压, 接近原色.
- **Dark (暗色)** — 20% 概率. 压低亮度 + 大幅脱色, 衣服偏暗灰.
- **Natural (自然)** — 8% 概率. 略带饱和的写实.
- **Vivid (鲜艳)** — 2% 概率. 高饱和创意色, 偶发.

风格档位决定了后续每个效果的参数范围和开启概率, 是整个随机逻辑的核心分支.

### 4.3 各效果的随机策略

**效果 1 — 色相/饱和度: 必开.**
色相在全范围 -180 到 180 度内随机. 饱和度和亮度按风格档分级:
- Realistic: 饱和度 [-25, 0], 亮度 [-8, 8]
- Dark: 饱和度 [-50, -20], 亮度 [-30, -10]
- Natural: 饱和度 [-15, 10], 亮度 [-10, 10]
- Vivid: 饱和度 [15, 40], 亮度 [-15, 15]

**效果 2 — 亮度/对比度: Dark 档强制开, 其他 30% 概率开.**
- Dark: 亮度 [-25, -5], 对比度 [5, 25] (压暗 + 提对比)
- Vivid: 亮度 [-12, 10], 对比度 [0, 25]
- 其他: 亮度 [-12, 10], 对比度 [-10, 12]

**效果 3 — 曲线: 15% 概率开** (刻意低概率, 避免与亮度/对比度叠加过度).
开启时生成一条 3 点 Master 曲线: 端点 (0,0) 和 (255,255) 固定, 中间控制点 x 在 [112, 144]、y 在 [108, 148] 范围内随机, 产生轻微的 S 型或反 S 型调整.

**效果 4 — 通道混合器: 永不开.** 原因: 容易产生怪异颜色, 不适合写实游戏.

**效果 5 — 颜色平衡: Realistic/Dark 30% 概率, 其他 50% 概率.**
三区 (阴影/中间调/高光) 各自随机选一个主轴 (R/G/B 之一), 在该轴上施加 ±[amp/2, amp] 范围的偏移, 其余两轴在 ±amp/4 范围内微调. amp 值: Vivid=22, Dark=8, 其他=12. 三区错开主轴防止共振染色. 保持亮度始终开启.

**效果 6 — 照片滤镜: Realistic/Dark 30% 概率, 其他 15% 概率.**
滤镜预设选择按风格分池:
- Dark: 仅从冷蓝/深色滤镜池 (冷色 80/LBB/82、深蓝、翡翠绿、深黄、水下) 中选
- Realistic: 从前 6 个暖/冷色预设中选
- 其他: 从全部 19 个预设中选
浓度: Dark 档 [20, 40]%, 其他 [10, 25]%. 保持亮度始终开启.

**效果 7 — 自然饱和度: Realistic/Dark 必开, 其他 30% 概率.**
用于在色相/饱和度之后再次微调饱和度水平:
- Realistic: vibrance [-15, 5], saturation [-20, -5] (进一步降饱和)
- Dark: vibrance [-25, -5], saturation [-30, -10] (大幅脱色)
- Natural: vibrance [-5, 15], saturation [-10, 5]
- Vivid: vibrance [5, 25], saturation [0, 15] (唯一增饱和的档位)

### 4.4 兜底机制

随机完成后检查是否至少有一个效果被开启. 若全部未开启 (理论上不会, 因为效果 1 必开, 但作防御), 则强制开启色相/饱和度, 色相随机 [-180, 180], 饱和度 [-15, 0].

### 4.5 影子保护不参与随机

`randomizeStack` 在进入时保存当前的 `shadowProtectThreshold`, 调用 `s.reset()` 清空全部效果后恢复该阈值. 影子保护始终保持用户手动设置的值, 不被随机覆盖.

## 五、随机触发入口与作用范围

随机功能通过 `ProjectController` (全局工程单例) 提供多个入口, UI 端在 `EffectPanel` 底部展示为一排按钮:

**随机当前层** (`randomizeCurrentLayer`): 仅对当前方案下当前选中的层调用一次 `randomizeStack`. 如果方案是本体 (isBuiltin) 或被锁定 (locked), 则跳过不执行.

**随机所有层** (`randomizeAllLayers`): 对当前方案下所有可见且非肤色保护的层各自独立调用 `randomizeStack`. 支持 `sameSeedAllLayers` 参数: 为 true 时所有层使用同一个随机种子, 产出"协调统一"的配色 (色相中心一致); 为 false 时每层独立种子, 各层完全独立. UI 默认传 false (各层独立).

**随机可编辑全部** (`randomizeAllSchemes(false)`): 遍历所有非本体方案, 跳过已烘焙 (isBaked) 和已锁定 (locked) 的方案, 对每个方案的所有可见非肤色层独立随机. 每方案每层使用独立种子 (seed=0).

**随机全部** (`randomizeAllSchemes(true)`): 同上, 但包含已烘焙方案. 已烘焙方案会被降级为可编辑状态 (isBaked 置 false, 原 add_lut PNG 引用清空), 然后参与随机. 此操作不可逆, UI 在执行前会弹出确认对话框提示用户已烘焙方案数量.

随机完成后, 如果方案数量 > 2, 会做一次 stable_partition 重排: 锁定的方案排在前面 (idx 1 起), 未锁定的排后面. 重排后自动更新方案名称中的编号前缀, 并重新定位当前选中方案.

所有随机操作都会设置 dirty 标记并发射 `effectsChanged` 信号, 触发预览刷新和缩略图重烘.

## 六、效果链 GPU 管线

效果链在 GPU 端的实现位于 `assets/shaders/effect_chain.hlsl`. 该 shader 定义了两个 Constant Buffer: `QuadCB` (register b0) 存储顶点 UV 和 NDC 矩形坐标, `EffectCB` (register b1) 存储全部 7 效果的参数.

`EffectCB` 的布局与 C++ 端的同名结构体严格对应: `enableMask.x` 是低 7 位的效果开关位掩码 (bit 0 = HSL, bit 1 = BrtCtr, ..., bit 6 = Vibrance), `enableMask.y` 是影子保护阈值. 其余字段按效果顺序排列, 每个效果占一个或多个 float4/int4.

核心函数 `applyEffectChain(float3 c0)` 按位掩码依次调用 7 个效果函数, 最后应用影子保护. 整个链在 sRGB 空间内完成 — RTV 使用 `DXGI_FORMAT_R8G8B8A8_UNORM` (非 `_SRGB`), shader 不做 sRGB↔linear 转换, blend 在 Gamma 空间执行, 与 AE 默认行为一致.

Pixel Shader 有两个入口点: `PSMain` 用于实时预览 (输入源帧, 输出变色后的帧, 保留 alpha), `PSMainBake` 用于离屏 LUT 烘焙 (输入 256x16 HALD-CLUT 颜色图, 输出变色后的 LUT 纹理).

顶点着色器 `VSMain` 使用 `SV_VertexID` 生成全屏三角形覆盖, 无需 IA buffer (6 个顶点, 2 个三角形).

渲染路径选择在 `FrameRenderer` 中进行: 若层有非 identity 的 `EffectStack`, 走 `effect_chain` PS (主路径); 若层绑定了 add_lut PNG 且标记为 useLut, 走 `recolor` PS (HALD-CLUT 反推路径); 否则走 `fullscreen_quad` 的透传 PS (本体).

## 七、LUT 烘焙管线

`LutBaker` (位于 `src/render/LutBaker.{h,cpp}`) 负责将 `EffectStack` 烘焙为一张 256x16 的 HALD-CLUT 纹理, 供导出和缩略图使用.

烘焙过程: 以内置的恒等 HALD-CLUT 颜色图 (`:/lut/default.png`, 即 `assets/颜色图.png`, 256x16) 作为源纹理, 绑定到 t0; 将 `EffectStack` 参数填入 `EffectCB` (cb1); 使用 `PSMainBake` 入口点在离屏 RenderTarget 上绘制一个全屏四边形. Shader 对颜色图的每个像素执行效果链变换, 输出到 256x16 RT.

RT 格式为 `R8G8B8A8_UNORM`, shader 直接写 sRGB 字节. 烘焙完成后通过 staging 纹理 (`D3D11_USAGE_STAGING`) 将结果回读到 CPU 端的 `m_lutBytes` 数组 (256x16x4 = 16384 字节). 回读后调用 `enforceShadowLock` 强制将 (0,0) 像素设为 (0,0,0), 确保纯黑不被浮点漂移影响.

当 `EffectStack.isIdentity()` 为 true 时, 理论上可以跳过 PS 直接拷源颜色图, 但实现中仍走管线 (em=0 时 PS 直接 return src), 因为 256x16 只需约 0.05ms, 优化收益不大.

烘焙出的 LUT 纹理可作为 SRV 绑定到 `recolor.hlsl` 的 t1 槽位, 供 HALD-CLUT 反推路径使用, 也可通过 `stb_image_write` 导出为 PNG 文件.

## 八、HALD-CLUT 颜色空间编码

本项目使用 16x16x16 的 HALD-CLUT, 平铺为 256x16 的 2D 纹理. 编码规则 (由 `assets/颜色图.png` 实证确认):

对于纹理上的像素 (x, y), 其中 x 属于 [0, 255], y 属于 [0, 15]:
- R_in = (x mod 16) * 17
- G_in = y * 17
- B_in = (x / 16) * 17

即 B 分量决定切片号 (x/16 取整, 共 16 个切片横向排列), R 分量是切片内的列 (x mod 16), G 分量是行 (y). 乘以 17 是因为 0-15 的 16 级映射到 0-255 的 256 级 (15 * 17 = 255).

运行时反推 (recolor.hlsl 的 `SampleHald` 函数): 将源像素 RGB 各乘 15 得到 [0, 15] 范围的浮点坐标. B 分量取 floor 和 ceil 得到相邻两个切片, 在切片间做线性插值. 切片内的 R/G 坐标直接映射到纹理 UV, 由 GPU 的双线性 sampler 自动处理 R/G 方向的插值. 最终效果等价于三线性查找.

`HaldClut` 类 (位于 `src/core/HaldClut.{h,cpp}`) 提供加载、验证和影子锁定功能. `isDefaultHaldEncoded` 方法可验证一张 256x16 PNG 是否符合默认编码 (容差 ±2). `enforceShadowLock` 将 RGBA 数组的第一个像素 (即 R=0, G=0, B=0 对应的输出) 强制设为 (0,0,0), 确保纯黑输入始终映射到纯黑输出.

## 九、曲线求解器

`CurveSolver` (位于 `src/core/CurveSolver.{h,cpp}`) 将用户编辑的曲线控制点编译为 256 字节的查找表, 供 GPU 纹理采样使用.

算法核心是 Fritsch-Carlson 单调三次 Hermite 插值, 与 Python 端实现一致. 流程为: 首先将控制点按 x 坐标排序; 若首点 x > 0 或末点 x < 255, 补上端点夹紧; 然后计算各段斜率和各控制点切线; 应用 Fritsch-Carlson 单调性修正 (当 `alpha^2 + beta^2 > 9` 时缩放切线); 最后对 0-255 的每个整数 x 用 Hermite 基函数求值.

`buildLut` 方法将 4 条曲线 (Master/R/G/B) 各自独立求解, 打包为 256x4 字节数组: 每行 4 字节, 分别是 Master(R通道)、R曲线(G通道)、G曲线(B通道)、B曲线(A通道). 该数组直接上传到 256x1 的 `R8G8B8A8_UNORM` 纹理 (`m_curveTex`), 绑定到 shader 的 t1 槽位.

## 十、方案与层级体系

### 10.1 方案 (Scheme)

项目支持多方案体系, 定义在 `src/core/Project.h` 的 `Scheme` 结构体. 方案 0 永远是"本体" (isBuiltin=true), 全层 disabled, 不可编辑效果. 用户可新增任意数量的方案, 每个方案为每一层独立维护一份 `EffectStack`.

方案有两种来源: 用户手编 (isBaked=false, `layerEffects` 存储 EffectStack, 渲染走 effect_chain PS) 和已烘焙导入 (isBaked=true, `layerLutPath` 存储 add_lut PNG 路径, 渲染走 recolor PS). 已烘焙方案不可编辑效果, 但可通过"随机全部"操作降级为可编辑.

方案支持锁定 (`locked` 标志): 锁住的方案不参与任何随机变色操作. 本体和已烘焙方案不受锁定影响 (有各自的保护机制).

### 10.2 层级 (Layer)

角色资源按层级组织: body (身体) → 00..04 (部件层) → addon (附件层). 每层在每个方案下可有不同的变色效果. 层级支持可见性切换和肤色保护 (`skinSafeLayerKeys`): 标记为肤色保护的层在所有方案下都不应用变色, 保持本体外观, 随机操作也会跳过这些层.

### 10.3 数据流

用户在 UI 上操作 EffectPanel 的滑块/曲线/开关 → 修改当前方案当前层的 `EffectStack` → 30ms debounce → `ProjectController::notifyEffectsChanged()` → 发射 `effectsChanged` 信号 → `FrameRenderer` 重新上传 `EffectCB` 到 GPU → 下一帧 D3D11 DrawCall 使用新参数渲染.

随机操作: 用户点击随机按钮 → `ProjectController` 调用 `randomizeStack` → 修改 `EffectStack` → `EffectPanel::rebuildForCurrentLayer()` 重建滑块 UI 反映新值 → 发射 `effectsChanged` 信号 → 同上.

LUT 导出: `LutBaker::bake(stack)` → 离屏渲染 → CPU 回读 → `stb_image_write` 写 PNG.

### 14.5 编译注意

| 现象 | 原因 | 处置 |
|------|------|------|
| `warning C4334: "<<": 32 位移位结果隐式转 64 位` (gif.h L400-401) | gif.h 内部 `1 << bitDepth` 在 64-bit 平台 warn | 可忽略 (不影响正确性). 若要清: `pragma warning(disable:4334)` |
| `#define GIF_MALLOC / GIF_FREE` 重定义 | gif.h 定义自己的 malloc/free | 在 include 前定义 `GIF_TEMP_MALLOC` / `GIF_TEMP_FREE` / `GIF_MALLOC` / `GIF_FREE` 为 `malloc/free` (已做) |
| 中文路径 fopen 失败 | gif.h 用 `fopen/fopen_s`, Windows 下 `fopen` 不支持 UTF-8 | 写到 `%TEMP%` 英文路径, 写完后 `QFile::copy` 到用户中文路径 (已做) |
| `stbi_image.h` 多重定义 (ODR violation) | gif.h 不依赖 stbi, 但如果同一 TU 包含两者需注意 `STB_IMAGE_IMPLEMENTATION` 仅定义一次 | GifExporter.cpp 只 `#include "gif.h"`, 不包含 stbi (已确保) |
