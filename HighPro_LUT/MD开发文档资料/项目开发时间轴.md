## 二、按阶段拆解 (开发日志视角)

> 计划 11~12 周, 实际 6 天 (05-30 → 06-04) 已交付可用静态 EXE.

### 阶段 0 · 立项前调研 (05-30 早)

**3 篇调研文档** (`颜色图处理.md` / `角色序列图处理.md` / `高性能序列帧角色换色工具EXE开发技术方案研究.md`):

- **用户起点**: 拿到一张 `颜色图.png` 和一套 9353 角色序列帧, 想做个 "类似 AE 调色但单 EXE 双击运行" 的工具
- **关键认知建立**:
  - `颜色图.png` 不是普通素材, 是 **HALD-CLUT 16³ 可视化** (256×16 = 4096 色)
  - AE 那 12 个效果可以砍到 **7 个核心** (反转 + 灰度/基值/增益 + 色阶 + 曝光度 砍掉, 因为可被曲线 / 亮度对比度 / 通道混合 覆盖)
  - 角色 = body + 0~4 数字层 + addon (单选), 共 7~8 个图层, 每层独立 EffectStack
- **框架对比结论**: Dear ImGui 不行 (UI 控件弱), Electron 不行 (体积+性能), 最终 **Qt 6 + D3D11**

### 阶段 1 · 立项 + 技术方案 (05-30 下午~晚)

`01_技术方案_HighPro_LUT.md` (22:52) 定稿:

| 决策 | 内容 |
|------|------|
| 技术栈 | C++17 / Qt 6 / Direct3D 11 / CMake / 全静态链接 |
| 颜色管线 | HALD-CLUT 16×16×16 (与 FFmpeg/GIMP/PS 兼容) |
| 7 效果链 | HSL → BrtCtr → Curves → ChMix → ColorBal → PhotoFilter → Vibrance (固定顺序) |
| 多方案 | Project.schemes 动态数组 + 画廊, 每方案独立 perLayer EffectStack |
| 影子保护 | LUT (0,0) 像素强制锁 (0,0,0) |
| UI 框架 | Qt6 Widgets + QDockWidget (左资源树 / 中双画布 / 右方案画廊 / 下效果手风琴) |
| 性能目标 | 4000 帧 60 FPS / 单帧变色 <1ms / 烘焙 <1ms / EXE 25~40MB / <2s 启动 |

**当晚产出**: `HighPro_LUT.exe` 76 KB, M1 D3D 清屏 #3C3C3C, 中文菜单, 双击不崩.

### 阶段 3 · 框架风险再确认 (06-01)

`Qt软件风险.md` (18:13): 用户立项后疑虑 — Qt 对大公司有什么风险?

**6 大风险类**:
1. 许可证风险 (商业 vs LGPL/GPL)
2. LGPL/GPL 合规难度高 (动态/静态链接 / 用户替换权)
3. 商业授权成本 + 谈判风险 (按开发者/产品线计费)
4. 供应商锁定 (信号槽 / MOC / RCC 深度绑定)
5. 版本策略 + 长期维护 (5~10 年 LTS / 升级牵动 SDK)
6. 开源治理 + 安全 + 人才池

**对本项目结论**: 单 EXE 桌面工具, 走 **LGPL 静态链接** 合规, 不依赖 WebEngine, 风险可控.

**同期源码**: M2 资源扫描 + 单层序列帧在写 (`PathUtil` / `LayerData` / `ResourceScanner` / `D3D11Texture` / `FrameLoader` / `FrameRenderer` 单层版).
