# 方案画廊新增功能开发计划：❤️细化方案 / 🔷配色方案转移

> 项目：`HighPro_LUT`  
> 模块：方案画廊 `SchemePanel`、项目控制器 `ProjectController`、效果数据 `EffectStack`  
> 目标：在方案画廊右键菜单新增并落地两个功能：`❤️细化方案`、`🔷配色方案转移`。

---

## 1. 当前代码现状判断

| 模块 | 当前能力 | 与新功能关系 |
|---|---|---|
| `src/ui/SchemePanel.cpp` | 方案画廊列表、右键菜单、重命名、复制、锁定、删除、缩略图刷新 | 两个新入口都应加在这里 |
| `src/core/Project.h` | `Scheme` 保存每个方案的 `layerEffects`、`layerLutPath`、`palette` | 细化方案与转移都要修改 `Scheme` 数据 |
| `src/core/ColorEffect.h` | `EffectStack` 已有 7 类效果，其中包含 `hsl` 与 `brtCtr` | 细化方案只开放这两个效果 |
| `src/app/ProjectController.*` | 负责方案增删改、随机、智能随机、信号刷新、dirty 标记 | 应新增“细化保存”和“方案层转移”接口 |
| `src/ui/EffectPanel.cpp` | 已有完整效果控件 UI，包含色相/饱和度、亮度/对比度等 | 可复用滑块控件思想，但细化弹窗要做独立轻量版 |
| `src/app/ProjectIO.cpp` | 已支持 `EffectStack` JSON 持久化 | 细化结果保存进 `layerEffects` 后可自然落盘 |

## 2. 功能一：❤️细化方案

### 2.1 用户流程

| 步骤 | 用户操作 | 系统行为 |
|---|---|---|
| 1 | 在方案画廊选中一个方案 | 当前方案作为细化目标 |
| 2 | 右键点击 `❤️细化方案` | 弹出独立细化窗口 |
| 3 | 在窗口内选择任意资源层 | 显示该层当前方案已有随机效果值 |
| 4 | 调整 `色相/饱和度`、`亮度/对比度` | 实时预览或保存后生效，按开发选择落地 |
| 5 | 点击保存 | 新参数合并进原方案对应层的 `EffectStack` |
| 6 | 关闭窗口 | 方案画廊缩略图、预览画布刷新 |

### 2.2 UI 设计

新增独立弹窗，建议文件：

| 文件 | 作用 |
|---|---|
| `src/ui/SchemeRefineDialog.h` | 细化弹窗声明 |
| `src/ui/SchemeRefineDialog.cpp` | 细化弹窗实现 |

弹窗布局：

| 区域 | 控件 | 说明 |
|---|---|---|
| 顶部 | 方案名标签 | 显示当前正在细化的方案，比如 `方案 8 - 02 变色` |
| 左侧 | 层级树/列表 | 使用 `Project.layers`，完整显示 `addon/*`、`00/01/02/...`、`body` |
| 右侧 | 效果控件 | 只显示两个分组：`色相/饱和度`、`亮度/对比度` |
| 底部 | `保存`、`取消`、可选 `重置当前层细化` | 保存后写回目标方案 |

效果控件范围沿用现有模型：

| 效果 | 参数 | 范围 | 对应字段 |
|---|---|---:|---|
| 色相/饱和度 | 色相 | `-180 ~ 180` | `EffectStack::hsl.hue` |
| 色相/饱和度 | 饱和度 | `-100 ~ 100` | `EffectStack::hsl.saturation` |
| 色相/饱和度 | 亮度 | `-100 ~ 100` | `EffectStack::hsl.lightness` |
| 亮度/对比度 | 亮度 | `-150 ~ 150` | `EffectStack::brtCtr.brightness` |
| 亮度/对比度 | 对比度 | `-100 ~ 100` | `EffectStack::brtCtr.contrast` |

### 2.3 数据继承规则

核心要求：细化不是覆盖整个随机结果，而是继承已有随机值，再叠加/修改指定效果。

建议落地规则：

| 场景 | 处理规则 |
|---|---|
| 目标方案已有 `layerEffects[layerKey]` | 复制现有 `EffectStack` 到弹窗临时缓存 |
| 目标方案没有该层 `EffectStack` | 创建默认 `EffectStack{}`，再允许细化 |
| 用户修改 HSL | 只改 `hsl` 字段，并开启 `enabled[EffectStack::EHsl] = true` |
| 用户修改亮度/对比度 | 只改 `brtCtr` 字段，并开启 `enabled[EffectStack::EBrtCtr] = true` |
| 其他 5 个效果 | 完全保留，不显示、不修改 |
| 点击取消 | 丢弃临时缓存，不写回项目 |
| 点击保存 | 将临时缓存写回目标方案 `layerEffects` |

### 2.4 已烘焙方案处理

当前项目存在两类方案：

| 类型 | 字段 | 当前行为 |
|---|---|---|
| 可编辑方案 | `isBaked=false` | 使用 `layerEffects` 实时渲染 |
| 已烘焙方案 | `isBaked=true` | 使用 `layerLutPath` 的 PNG LUT |

细化功能需要改效果栈。已烘焙方案如果要细化，必须转为可编辑。

建议规则：

| 情况 | 处理 |
|---|---|
| 选中本体方案 `isBuiltin=true` | 禁用 `❤️细化方案` |
| 选中可编辑方案 | 直接进入细化 |
| 选中已烘焙方案 | 弹确认：会降级为可编辑，原 `add_lut` 引用清空；确认后继续 |

### 2.5 预览策略

| 方案 | 优点 | 缺点 | 建议 |
|---|---|---|---|
| 保存后预览 | 简单、安全 | 调参时不能实时看结果 | 第一版采用 |
| 实时预览 | 体验好 | 需要临时应用与回滚，状态复杂 | 第二版优化 |

第一版建议：窗口内只调参数，点击保存后刷新预览和缩略图。

## 3. 功能二：🔷配色方案转移

### 3.1 用户流程

| 步骤 | 用户操作 | 系统行为 |
|---|---|---|
| 1 | 在方案画廊选中源方案，例如方案 8 | 源方案为当前右键所在方案 |
| 2 | 右键 `🔷配色方案转移` | 出现层级子菜单 |
| 3 | 选择资源树层级，例如 `addon` | 继续展开目标方案 ID 菜单 |
| 4 | 选择目标方案 ID，例如 `5` | 将源方案的 `addon` 配色数据复制到目标方案 5 的 `addon` |
| 5 | 系统刷新 | 目标方案缩略图、画布、项目 dirty 状态更新 |

### 3.2 菜单结构

右键菜单结构建议：

```text
🔷 配色方案转移
  ├─ addon
  │   ├─ 1
  │   ├─ 2
  │   ├─ ...
  │   └─ 27
  ├─ 01
  │   ├─ 1
  │   ├─ 2
  │   └─ ...
  ├─ 00
  │   ├─ 1
  │   └─ ...
  └─ body
      ├─ 1
      └─ ...
```

目标方案 ID 来自 `Project.schemes` 当前已有方案，不写死 1~27。  
层级来自 `Project.layers` 当前资源树结构，不写死 `addon/00/01/body`。

### 3.3 层级匹配规则

需求强调：`addon` 可能很多，`00 01 02 03 04 05 06... body` 需要跟资源树走。

建议定义“转移分组”：

| 分组显示 | 包含层 | 说明 |
|---|---|---|
| `body` | `LayerKind::Body` | 本体层 |
| `00`、`01`、`02`... | `LayerKind::Numbered` 且 `numberedIdx` 对应 | 数字层，未来扩展到 `05/06/...` 自动支持 |
| `addon` | 所有 `LayerKind::Addon` | 一次转移所有 addon 子层；若后续需要，也可展开 `addon/01`、`addon/02` |

第一版建议：`addon` 作为组，转移所有 addon 子层。这样符合截图中只显示 `addon` 的交互。

### 3.4 转移数据规则

“把现在的 ID 颜色方案给到被选中的 ID 方案上”本质是源方案某一层/一组层的颜色数据复制到目标方案对应层。

建议规则：

| 源方案类型 | 目标方案类型 | 转移内容 | 处理 |
|---|---|---|---|
| 可编辑 | 可编辑 | `layerEffects[layerKey]` | 直接复制 |
| 可编辑 | 已烘焙 | `layerEffects[layerKey]` | 目标降级为可编辑，清空 `layerLutPath` 后复制 |
| 已烘焙 | 已烘焙 | `layerLutPath[layerKey]` | 复制对应 LUT 路径，目标仍可保持已烘焙 |
| 已烘焙 | 可编辑 | 源缺 `layerEffects` | 建议弹提示：已烘焙源不支持转为效果栈，或目标改为已烘焙局部路径模式会破坏现架构 |

为了和当前渲染架构简单兼容，建议第一版采用以下原则：

1. 如果源方案是可编辑：复制 `EffectStack`。
2. 如果源方案是已烘焙：复制 `layerLutPath`，但仅允许目标方案也作为已烘焙/降级前确认。
3. 避免一个方案同时半边 `layerEffects`、半边 `layerLutPath` 的混合状态，除非后续专门改渲染管线支持混合。

更稳的第一版落地：

| 条件 | 行为 |
|---|---|
| 源方案 `isBaked=false` | 支持转移到任意非本体目标；目标若 `isBaked=true`，确认后降级为可编辑 |
| 源方案 `isBaked=true` | 支持转移到 `isBaked=true` 目标；转移 LUT 路径 |
| 源方案 `isBaked=true` 且目标可编辑 | 暂不支持，提示先复制为新方案或重新随机生成可编辑效果 |

### 3.5 示例

需求示例：把方案 8 的 `addon` 随机颜色方案给到方案 5 的 `addon`。

执行后：

| 项 | 值 |
|---|---|
| 源方案 | `schemes[8]` |
| 目标方案 | `schemes[5]` |
| 分组 | `addon` |
| 实际复制层 | 所有 `LayerKind::Addon` 的 `layer.key()` |
| 写入位置 | `schemes[5].layerEffects[addonLayerKey] = schemes[8].layerEffects[addonLayerKey]` |
| 刷新 | `effectsChanged()`、`schemesChanged()`、缩略图重烘 |

## 5. ProjectController 建议新增接口

```cpp
// 打开细化前，确保方案可编辑。已烘焙可选择降级。
bool ensureSchemeEditable(int schemeIdx, bool allowConvertBaked, QString* errorOut = nullptr);

// 保存细化弹窗临时结果。
bool applyRefinedLayerEffects(int schemeIdx,
                              const QHash<QString, EffectStack>& refinedEffects,
                              QString* errorOut = nullptr);

// 复制一个分组的配色数据：sourceSchemeIdx -> targetSchemeIdx。
bool transferSchemeColorGroup(int sourceSchemeIdx,
                              int targetSchemeIdx,
                              const QString& groupKey,
                              QString* errorOut = nullptr);
```

`groupKey` 建议值：

| groupKey | 含义 |
|---|---|
| `body` | body 层 |
| `num:00`、`num:01` | 数字层 |
| `addon` | 所有 addon 层 |

## 8. 风险与处理

| 风险 | 说明 | 处理建议 |
|---|---|---|
| 已烘焙方案无法反推 `EffectStack` | PNG LUT 不是效果参数，不能可靠还原 HSL/Brightness | 已烘焙细化前降级，或提示限制 |
| 方案混合 `layerEffects` 与 `layerLutPath` | 当前 `Scheme` 用 `isBaked` 区分两套渲染路径，混合会复杂 | 第一版避免混合状态 |
| addon 层级很多 | 菜单可能过长 | 第一版 addon 作为组；第二版可展开 `addon/01` |
| 目标方案锁定 | 用户语义是保护方案 | 默认禁用转移到锁定方案，或二次确认 |
| 细化实时预览回滚复杂 | 弹窗改动要临时写入再撤销 | 第一版保存后生效 |

---

## 9. 推荐落地顺序

1. 先做 `❤️细化方案` 保存后生效版本。
2. 再做 `🔷配色方案转移` 可编辑方案之间的 `EffectStack` 复制。
3. 最后补已烘焙方案处理与确认弹窗。
4. 第二轮优化实时预览、addon 子层精细选择。

## 10. 第一版完成定义

第一版开发完成后，应满足：

- 方案画廊右键菜单出现两个新增功能。
- `❤️细化方案` 能打开独立窗口。
- 细化窗口能选择所有资源层。
- 细化窗口只显示 `色相/饱和度` 与 `亮度/对比度`。
- 保存细化后，修改写回原方案随机值所在 `EffectStack`，其他效果不丢。
- `🔷配色方案转移` 子菜单按资源树动态生成层级。
- 目标方案 ID 按当前已有方案动态生成。
- 能完成“方案 8 的 addon 配色转移到方案 5 的 addon”。
- 操作后预览与方案缩略图刷新。
- 工程保存/打开后结果保留。
